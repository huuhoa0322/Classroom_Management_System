Agile User Stories & Acceptance Criteria

Classroom Management System (CLS) - Project Requirements

Phase: MVP (3-Weeks) Version: 0.1 Format: BDD (Behavior-Driven Development) Standard
Ticket ID: CLS-001: Onboard New Student Profiles Student Management
User Story

As an Academic Admin,
I want to onboard new student profiles and record their demographic and parent information,
So that I can accurately track their lifecycle centrally and ensure Parent contact data is ready for zero-touch notification operations.

Acceptance Criteria
AC1: Successful Student Onboarding
Given I am on the "Add New Student" page When I enter valid demographic inputs and associate the student with a valid Parent Email and Phone Number Then the system must successfully save the new student profile And link the record directly to the Parent entity for future automated communications.
AC2: Missing Critical Parent Email
Given I am creating a new student profile When I attempt to save the form while leaving the "Parent Email" field blank Then the system must prevent the submission And highlight the field in red with an error message: "Parent Email is strictly required to enable zero-touch automated notifications."
Ticket ID: CLS-002: Update Student Lifecycles Student Management
User Story

As an Academic Admin,
I want to edit existing student records and manage their lifecycle status (Active, Dropped Out),
So that the central database accurately reflects active enrollments and prevents inactive students from appearing in scheduling blocks.

Acceptance Criteria
AC1: Deactivating a Dropped-out Student
Given a student profile is currently in an "Active" state When I change their enrollment status to "Dropped Out" and save the record Then the system must completely archive their active tuition packages securely And remove the student from any future uncommenced class sessions automatically.
Ticket ID: CLS-003: Record Offline Tuition Payments Financial Administration
User Story

As an Academic Admin,
I want to log offline tuition fee collections and allocate purchased packages,
So that I can monitor accurate financial records, track remaining sessions automatically, and mitigate the risk of bad debt compared to spreadsheet tracking.

Acceptance Criteria
AC1: Manual Payment Entry and Package Allocation
Given I am on a student's financial dashboard When I record a payment transaction of a certain amount and select a predefined "30-Session Tuition Package" Then the system must log the payment timestamp and method And instantly credit 30 sessions to the student's active Student_Package balance.
Ticket ID: CLS-004: Setup Core Teaching Schedules Schedule Management
User Story

As an Academic Admin,
I want to define master class timetables by allocating specific Rooms and assigning verified Teachers,
So that the center's resources are organized efficiently and systematically recorded in the central database.

Acceptance Criteria
AC1: Session Allocation Success
Given I am viewing the Schedule Maker tool When I select a Class, specify an available Room, assign an available Teacher, and configure a valid Date/Time matching the center's operating hours Then the system must create the new session block successfully And display it across the master timetable grid.
Ticket ID: CLS-005: Resolve Scheduling Conflicts Schedule Management
User Story

As an Academic Admin,
I want the system to strictly validate and block any double-booking attempts,
So that we achieve the goal of 0 scheduling conflicts, eliminating frustrating resource overlaps for operations and teachers.

Acceptance Criteria
AC1: Prevent Teacher Double Booking
Given "Teacher A" is already assigned to an active session from 18:00 to 19:30 When I attempt to create or edit another session assigning "Teacher A" passing through the 18:00 - 19:30 timeframe Then the system must definitively block the action And prompt an error specifically reading: "Scheduling Conflict: This Teacher is already assigned to a session in the selected time range."
AC2: Prevent Physical Room Overlaps
Given "Room A" is already assigned to a session from 14:00 to 16:00 When I attempt to schedule a new session allocating "Room A" at 15:00 to 17:00 Then the system must block the action And prompt an error specifically reading: "Scheduling Conflict: This Room is occupied during the selected time window."
Ticket ID: CLS-006: Review Package Renewal Alerts Retention Management
User Story

As an Academic Admin,
I want to view a centralized dashboard of system-triggered package depletion alerts,
So that I can actively intervene and consult Parents for package renewals 2-weeks ahead of expiry, subsequently reducing the center's churn rate by 20%.

Acceptance Criteria
AC1: View Depletion Targets
Given the automated System Timer has flagged relevant students with expiring packages When I access the "Renewal Alerts" module Then I must see a tabulated, sortable list containing the Student Name, Parent Contact, and Remaining Sessions/Days And be able to toggle an alert status marker confirming I have "Consulted" the parent.
Ticket ID: CLS-007: View Personalized Timetables Academic Operations
User Story

As a Teacher,
I want to access a personalized, mobile-friendly view of my daily or weekly teaching rosters,
So that I can seamlessly prepare for my lessons without experiencing friction or asking Admin for manual Excel snapshots.

Acceptance Criteria
AC1: Mobile Access to Roster
Given I am securely logged in as a Teacher via a mobile browser When I launch the "My Roster" tab Then I must view a chronological list of my assigned upcoming sessions And see the Class Name, Start/End Times, and assigned Room clearly.
Ticket ID: CLS-008: Record Class Attendance Academic Operations
User Story

As a Teacher,
I want to quickly mark students as Present, Absent, or Late directly through a simplified digital flow,
So that the attendance data propagates immediately to deduct from the student's tuition package balance and hooks into the zero-touch Parent email system.

Acceptance Criteria
AC1: Immediate Attendance Propagation
Given I open the attendance sheet for a completed session When I change statuses for students to "Present" or "Absent" and successfully submit the sheet Then the system must update the attendance repository And immediately deduct 1 session duration automatically from the package balance of any student marked "Present" or "Late".
Ticket ID: CLS-009: Submit Academic Performance Feedback Academic QA
User Story

As a Teacher,
I want to submit qualitative performance feedback or scores securely within a 12-hour post-session SLA,
So that I adhere to center operating rules, ensuring Parents receive timely, valuable academic evaluations.

Acceptance Criteria
AC1: Feedback within Valid SLA Window
Given a session formally completed less than 12 hours ago When I submit a qualitative text observation via the feedback tool Then the system strictly accepts and finalizes the submission.
AC2: SLA Infraction Monitoring
Given a session completed over 12 hours ago When I access the session to submit a late feedback observation Then the system applies a metadata tag labeling it an "SLA Overdue Violation" And accepts the late entry while capturing the infraction sequence for the Admin QA audit.
Ticket ID: CLS-010: Execute Daily Depletion Scans System Automation
User Story

As the System Timer,
I want to execute automated cron jobs daily to scan through active student package balances,
So that I can securely identify near-exhausted packages and populate the Renewal Alerts module without human intervention.

Acceptance Criteria
AC1: Threshold Detection Job
Given the system time reaches 00:00 (Midnight) When the backend cron job initiates a table scan querying all active Student_Package balances Then any active student returning 4 or fewer remaining sessions must trigger the creation of a "Renewal Alert" record assigned to the Academic Admin module.
Ticket ID: CLS-011: Dispatch Zero-Touch Parent Notifications System Automation
User Story

As an Email Gateway API,
I want to receive data payloads corresponding to crucial lifecycle milestones (Attendance tracking, Schedule shifts),
So that I can instantly deliver branded, zero-touch confirmation emails to the Parent's inbox, fostering trust and operational transparency.

Acceptance Criteria
AC1: Attendance Confirmation Trigger
Given the gateway acknowledges a "submit attendance" internal event trigger from a Teacher When the attendance payload is securely received Then the gateway automatically formats and dispatches a notification Email detailing the session date, the attendance status, and the newly calculated remaining package balance to the primary Parent Email on file.
AC2: Schedule Change Broadcasting
Given the gateway detects a "Session Modified" event (e.g. Time or Date shifted) performed by the Academic Admin When the event payload finishes parsing Then the gateway dispatches an alert Email directly to all associated parents illustrating the schedule adjustment.
