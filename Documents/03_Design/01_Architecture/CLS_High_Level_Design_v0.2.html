1.1 Software Architecture

Kiến trúc Phần mềm Tổng thể — Modular Monolith (Layered Architecture)

The CLS system adopts a Modular Monolith architecture following the Layered Architecture pattern (3-project solution) as defined in ADR-001. This combination provides domain-centric design with clear bounded contexts while maintaining a single deployable unit — optimized for the project's 3-week MVP timeline, lean operational scale (1 branch, 50-150 learners), and a small development team (1 Tech Lead + 2-4 developers).

1.1.1 Architecture Principles

Layered Architecture (3-Tier) Strict top-down dependency flow: API depends on Business Logic Layer (BLL), and BLL depends on Data Access Layer (DAL). Entities are defined within the DAL to facilitate direct ORM mappings.
Module Encapsulation Each business domain (Student, Scheduling, Tuition, etc.) is encapsulated with its own Services, DTOs, Entities, and Validators. Inter-module calls only through service interfaces.
Shared Kernel Common base entities (AuditableEntity), enums, and shared value objects are centralized in Domain.Common to ensure consistency across all modules.
Stateless Communication REST API with JWT Bearer authentication via ASP.NET Identity. No server-side sessions. Horizontal scalability without sticky sessions.

1.1.3 Layer Responsibilities

Layer / Project Technology Responsibility
Presentation React 19, Vite 8, Tailwind CSS 4, Shadcn/ui, TanStack Query, TypeScript Renders the user interface (SPA). Handles client-side routing, form validation (React Hook Form + Zod), server state management (TanStack Query), and API communication via Axios with JWT token interceptors. Deployed separately on Vercel/Netlify.
CLS.API ASP.NET Core 10, Kestrel, ASP.NET Identity, JWT Bearer HTTP entry point. Hosts REST Controllers, JWT authentication middleware, CORS policy, action filters, global exception handling, and Swagger UI documentation. Orchestrates DI registration for all layers.
CLS.BLL C# Services, FluentValidation, AutoMapper Contains all domain-specific business logic organized into bounded contexts. Each module owns its Services, DTOs, Validators, and Mapping Profiles. Directly references CLS.DAL.
CLS.DAL EF Core 10, PostgreSQL 15+, Npgsql.EntityFrameworkCore.PostgreSQL Provider, SendGrid Implements data access (AppDbContext, Repositories), EF Core Migrations, entity type configurations (Fluent API), and external service integrations (email dispatch via SendGrid/Amazon SES).

Architecture Decision Reference: This design strictly adheres to ADR-001: Adopt Modular Monolith with ASP.NET + React Tech Stack. The modular monolith was selected over Microservices (rejected — over-engineered for 50-500 CCU) and Serverless (rejected — cold-start latency and vendor lock-in). Layered Architecture's 4-project structure (API → BLL → DAL) enforces dependency inversion by design, with module boundaries reinforced via internal access modifiers in .NET project references.

1.2.2 Project Dependency Rules (Layered Architecture)

# Rule Rationale
1 CLS.DAL has zero local project references Data Access Layer and Entities contain no local dependencies. Contains core Entities, DbContext, and Repositories.
2 CLS.BLL references only CLS.DAL Business logic depends on DAL\'s entities and repository interfaces.
4 CLS.API references CLS.BLL and CLS.DAL Orchestrates DI registration. Hosts Controllers that consume BLL services.
5 Use internal access modifier for repository implementations .NET enforces module boundaries at the project level. Repository classes in Infrastructure are internal — only exposed via interface registration in DI container.
6 Frontend features/ mirror backend modules Reduces cognitive overhead. Developers can locate frontend code for any backend module instantly (e.g., features/student/ → Student module).

1.2.3 Inter-Module Dependency Matrix

Source Module Target Module Dependency Type Description
Scheduling Course Service interface Schedule creation requires class and teacher lookup for room/teacher assignment and conflict validation.
Attendance Student Service interface Check-in requires student lookup. Session counter decrements student's remaining package sessions.
Attendance Notification Service interface Attendance events trigger automated email dispatch to parent's registered email address.
Tuition Student Service interface Payment tracking is linked to student enrollment. Package depletion alert requires student's remaining sessions.
Tuition Notification Service interface Renewal alert (2-week warning) triggers email to both Admin and Parent via notification engine.
Feedback Scheduling Service interface Teacher feedback is submitted post-session. Feedback references the specific schedule entry (class + time slot).

Design Constraint: All inter-module dependencies are unidirectional and flow through service interfaces defined in CLS.BLL. Circular dependencies between modules are strictly prohibited. The .NET project reference system provides compile-time enforcement — if a circular reference is introduced, the build will fail immediately.
